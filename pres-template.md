# Presentation Slide Template

A comprehensive template for defining professional presentation slides with complete speaker notes, background research, and implementation guidance. This template captures all elements needed to generate consistent, well-structured presentation content.

---

## Template Structure Overview

Each slide definition contains the following major sections:

1. **Slide Header** - Basic identification and classification
2. **Content** - The visible slide content
3. **Graphic/Visual** - Description of visual elements
4. **Speaker Notes** - What the presenter says
5. **Background** - Research and rationale
6. **Sources** - Citations and references
7. **Implementation Guidance** - Practical application advice

## SLIDE TYPE DEFINITIONS

Use these type classifications in the slide header to indicate the slide's role in the presentation:

### **TITLE SLIDE**
- Opens the presentation
- Contains: Title, Subtitle, Presenter info, Date
- Graphic: Hero image or thematic illustration
- Speaker notes focus on opening hook and setting expectations

### **PROBLEM STATEMENT**
- Establishes the challenge or pain point
- Creates tension that the presentation will resolve
- Often includes statistics or research data
- Speaker notes should create emotional resonance

### **INSIGHT / REVELATION**
- Delivers a key insight or "aha moment"
- Reframes how audience thinks about the problem
- Often contrasts common misconception with reality
- Speaker notes should build to and land the insight

### **CONCEPT INTRODUCTION**
- Introduces a new term, framework, or mental model
- Provides clear definition and context
- May include analogy for comprehension
- Speaker notes should explain why this matters

### **FRAMEWORK / MODEL**
- Presents a structured approach or methodology
- Often uses diagrams, pillars, or numbered components
- Shows relationships between elements
- Speaker notes walk through each component

### **COMPARISON**
- Contrasts two or more approaches, options, or states
- Uses tables, side-by-side layouts, or before/after
- Highlights key differentiators
- Speaker notes explain implications of differences

### **DEEP DIVE**
- Provides detailed exploration of a specific topic
- May include technical content, code, or specifications
- Supports the main argument with depth
- Speaker notes can be more technical

### **CASE STUDY**
- Presents a real-world example or application
- Includes specific outcomes, metrics, or quotes
- Validates the presentation's claims
- Speaker notes tell the story narratively

### **PATTERN / BEST PRACTICE**
- Describes a proven approach or methodology
- Often includes do's and don'ts
- May include pseudocode or implementation details
- Speaker notes explain why the pattern works

### **METRICS / DATA**
- Presents quantitative information
- Uses charts, graphs, or data tables
- Supports claims with evidence
- Speaker notes interpret the data for audience

### **ARCHITECTURE / DIAGRAM**
- Shows system structure or process flow
- Uses visual representation as primary content
- Labels and callouts explain components
- Speaker notes walk through the diagram

### **OBJECTION HANDLING**
- Anticipates and addresses audience concerns
- Presents objection-response pairs
- Builds confidence in the proposed approach
- Speaker notes should feel conversational

### **ACTION / NEXT STEPS**
- Provides concrete actions for the audience
- Often time-bound (this week, 30 days, 90 days)
- Makes the path forward clear
- Speaker notes should be motivating

### **SUMMARY / RECAP**
- Consolidates key points from a section
- Reinforces main messages
- Prepares for transition to next section
- Speaker notes should feel like natural conclusion

### **SECTION DIVIDER**
- Marks transition between major sections
- Minimal content - usually just section title
- Provides mental break for audience
- Speaker notes briefly preview upcoming section

### **CLOSING / CALL TO ACTION**
- Final slide before Q&A
- Summarizes core thesis
- Includes clear call to action
- Speaker notes should be memorable and inspiring

### **Q&A / CONTACT**
- Invites audience questions
- Includes contact information
- May include resource links
- Speaker notes: "Questions?"

### **APPENDIX**
- Supplementary material not in main flow
- Detailed data, extended examples, or references
- Available if needed during Q&A
- Minimal speaker notes

## SPEAKER NOTES CONVENTIONS

### Stage Directions (in brackets)
- `[Pause]` - Deliberate silence for effect
- `[Pause for effect]` - Longer pause after key statement
- `[Let that land]` - Allow insight to sink in
- `[Point to X]` - Gesture to specific visual element
- `[Emphasize this]` - Vocal emphasis on following statement
- `[Light humor]` - Delivery should be lighter
- `[Personal story - adjust to your context]` - Placeholder for presenter customization
- `[Show of hands]` - Audience participation cue
- `[Look around room]` - Connect with audience

### Transition Markers
- `[Transition]` - Standard transition cue
- `[Transition: Click to next slide]` - Explicit click instruction
- `[OPENING - Description]` - Opening segment marker
- `[Hook - Description]` - Attention-grabbing opener

### Timing Guidance
- Include approximate time markers for longer presentations
- Note sections that can be shortened if running long
- Mark optional deep-dive content

### Audience Engagement
- Rhetorical questions that don't expect answers
- Actual questions with `[Wait for responses]`
- Acknowledgment of likely audience experience
- References to earlier audience input

### Key Research & Citations (3-5 entries)
Format: **[Source Name (Year)]**: [Detailed explanation]

- Cite specific statistics or findings

## EXAMPLE: COMPLETE SLIDE DEFINITION

```markdown
---

## **SLIDE 7: CONCEPT INTRODUCTION**

**Title**: Memory Lives in Structure, Not Agents

**Content**:

**Core Principle:**
**Agents should be stateless. Memory should be external and structured.**

Think of it this way:
- ❌ Trying to give agents memory
- ✓ Building systems where amnesiacs can work effectively

**Analogy:**
A factory where you hire a new worker every day, but they succeed because:
- Clear task lists show what's done/pending
- Logs show what's been tried
- Procedures show how to validate work
- Records show what to do next

**Graphic**: Two side-by-side illustrations. Left: Agent with thought bubble trying to "remember" (crossed out, muted colors). Right: Agent reading from clipboard/dashboard showing project status (checkmark, bright colors). The contrast emphasizes external vs. internal memory.

**SPEAKER NOTES**:

[INSIGHT - Core architectural principle]

"Here's the insight that took me way too long to internalize, and once you get it, everything else makes sense:

[Pause for effect]

Stop trying to give agents memory. Instead, build systems where amnesiacs can work effectively.

[Let that land]

This sounds like a subtle distinction, but it's the difference between systems that fail and systems that work.

Think about it this way—imagine you run a factory, but for some weird reason, you have to hire a completely new worker every single day. They have skills, they have tools, but they have zero memory of what happened yesterday.

That sounds impossible, right? But actually, we know exactly how to make this work:

[Point to each bullet]
- You give them a task list showing what's completed and what's pending
- You give them logs showing what's been tried and what failed
- You give them procedures for validating their work
- You give them records showing exactly what to do next

The worker is still an amnesiac. But the FACTORY has memory. The system has memory. And because that memory is external and structured, any worker can walk in and be productive immediately.

[Point to graphic on right] That's the model. The agent doesn't remember. The agent READS. It reads from structured records that tell it everything it needs to know."

[Transition]

"Now, before we go further, let me clarify something important..."

**BACKGROUND:**

**Rationale:**
- This slide introduces the core architectural insight that transforms agent design thinking
- The stateless agent pattern aligns with proven software engineering principles (stateless microservices)
- The factory worker analogy bridges abstract AI concepts to familiar organizational patterns
- This principle is endorsed by Anthropic, Google ADK, and 12-Factor Agents methodology

**Key Research & Citations:**
- **12-Factor Agents (HumanLayer, 2025)**: Factor 6 "Launch/Pause/Resume" and Factor 12 "Stateless Reducer" explicitly advocate for agents that can restart from structured state without losing context.
- **Anthropic Building Effective Agents (2025)**: Recommends separating "initializer agents" from "coding agents" with structured handoff between them.
- **Software Engineering Precedent**: The stateless service pattern from microservices architecture demonstrated that externalizing state enables horizontal scaling and fault tolerance.

**Q&A Preparation:**
- *"Doesn't this add complexity?"*: Initially, yes. But it's the same trade-off as stateless microservices vs. stateful monoliths. The upfront investment pays dividends in reliability and debuggability.
- *"What about conversation memory features?"*: Those are helpful for personal assistants but insufficient for production workflows. They're opaque, unstructured, and not designed for work coordination.
- *"How is this different from logging?"*: Logs are write-only records for debugging. Domain memory is structured, queryable state designed to guide agent behavior.

**Sources:**
1. [12-Factor Agents](https://github.com/humanlayer/12-factor-agents) - Foundational methodology for production agents
2. [Anthropic: Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) - Official guidance on agent architecture
3. [The Twelve-Factor App](https://12factor.net/) - Original stateless service principles
4. [Google ADK Overview](https://google.github.io/adk-docs/) - Multi-agent framework with tiered memory

**IMPLEMENTATION GUIDANCE:**

**Getting Started:**
- Identify what state your agent currently holds implicitly
- Design a minimal schema for externalizing this state (JSON/YAML files)
- Create a "handoff document" template capturing everything a new agent needs

**Best Practices:**
- Treat each agent invocation as potentially the first one
- Use structured formats (JSON, YAML) rather than prose for state persistence
- Include timestamps and versioning in persisted state
- Separate "facts" from "interpretations"—let the agent re-interpret fresh

**Common Pitfalls:**
- Storing too little state (can't resume) or too much (unwieldy)
- Failing to validate state integrity before agent resumes
- Designing state for humans rather than for agents to act on

**Pseudocode Pattern - Stateless Agent with External Memory:**
```python
class StatelessAgent:
    def __init__(self, memory_store):
        self.memory = memory_store  # External, not internal

    def run(self, task_id: str):
        # 1. LOAD: Read current state from external memory
        state = self.memory.load(task_id)

        # 2. ORIENT: Understand where we are
        current_task = state.get_next_incomplete_task()

        # 3. EXECUTE: Do exactly one task
        result = self.execute_single_task(current_task)

        # 4. UPDATE: Write results to external memory
        state.record_result(current_task, result)
        self.memory.save(task_id, state)

        # 5. STOP: Agent is disposable, memory persists
        return result
```

**Tools & Technologies:**
- **File-based state**: CLAUDE.md or NOTES.md files for single-agent workflows
- **Database-backed state**: PostgreSQL/SQLite for complex multi-agent systems
- **Agent frameworks**: LangGraph checkpointing, CrewAI memory systems

---
```

## USAGE NOTES

1. **Consistency**: Use the same structure for every slide to maintain consistency
2. **Completeness**: Fill in all sections, even if brief - this ensures comprehensive coverage
3. **Adaptation**: Adjust section depth based on slide complexity and importance
4. **Cross-referencing**: Reference other slides by number when building on previous content
5. **Iteration**: Treat slide definitions as living documents - refine based on feedback

This template ensures that presentations are:
- **Reproducible**: Another author can create similar quality content
- **Comprehensive**: No important elements are forgotten
- **Actionable**: Audience leaves with concrete next steps
- **Defensible**: Claims are backed by research and sources
- **Deliverable**: Speaker has everything needed to present confidently
